# 项目执行流程说明

本文档说明当前仓库从 C kernel 到 MLIR、Arm SME、LLVM IR、汇编和可执行文件的完整执行流程。

根目录 [README.md](/Users/alpaca/Documents/SME/SME1/README.md) 主要描述研究方案；本文档主要描述如何实际运行当前仓库中的各个步骤。

## 1. 环境约定

当前实验默认使用本机已经配置好的 LLVM / MLIR 工具链：

```text
/Users/alpaca/Documents/SME/external/llvm-project/build/bin
```

常用工具包括：

- `mlir-opt`
- `mlir-translate`
- `llc`
- `clang`
- `cmake`

其中第 04 步和第 05 步会使用 Arm SME 相关 lowering / codegen，因此 `llc` 阶段需要使用：

```bash
-mattr=+sme
```

## 2. 总体主线

当前仓库主线如下：

```text
gemm_mlir_kernel.c
-> 01_custom_lifter/output/gemm_fp32_linalg.mlir
-> 02_prefetch_cost_model/output/prefetch_analysis.json
-> 03_prefetch_injection/output/02_vector_prefetch.mlir
-> 04_vector_arm_sme_llvm/output/02_llvm_prefetch.mlir
-> 05_native_build_run/output/step4_llvm_prefetch.ll
-> 05_native_build_run/output/step4_llvm_prefetch.s
-> 05_native_build_run/output/step4_llvm_prefetch_demo
```

各步骤职责是：

| 步骤 | 目录 | 主要作用 | 关键输出 |
| --- | --- | --- | --- |
| 01 | `01_custom_lifter` | 将 C kernel 提升为高层 `linalg` MLIR | `gemm_fp32_linalg.mlir` |
| 02 | `02_prefetch_cost_model` | 转成 affine-normalized 分析层并生成预取决策 | `prefetch_analysis.json` |
| 03 | `03_prefetch_injection` | 转成 vector 层并插入 `memref.prefetch` | `02_vector_prefetch.mlir` |
| 04 | `04_vector_arm_sme_llvm` | 从第三步 vector 预取结果降到 Arm SME / LLVM dialect | `02_llvm_prefetch.mlir` |
| 05 | `05_native_build_run` | 导出 `.ll`，生成汇编、目标文件和可执行文件 | `step4_llvm_prefetch.s` / demo |
| 06 | `06_experiment_validation` | 当前已清空，预留后续实验验证 | `output/.gitkeep` |

## 3. 第 01 步：C kernel 提升到 linalg MLIR

输入：

- [gemm_mlir_kernel.c](/Users/alpaca/Documents/SME/SME1/gemm_mlir_kernel.c)

执行：

```bash
python3 01_custom_lifter/lift_gemm.py gemm_mlir_kernel.c
```

输出：

- [01_custom_lifter/output/gemm_fp32_linalg.mlir](/Users/alpaca/Documents/SME/SME1/01_custom_lifter/output/gemm_fp32_linalg.mlir)

验证：

```bash
/Users/alpaca/Documents/SME/external/llvm-project/build/bin/mlir-opt \
  01_custom_lifter/output/gemm_fp32_linalg.mlir
```

这一层的核心是 `linalg.matmul`。它保留了矩阵乘的高层语义，并作为后续 affine 分析和 vector lowering 的共同入口。

## 4. 第 02 步：构建预取 Cost Model

构建 pass：

```bash
cmake -S 02_prefetch_cost_model \
  -B 02_prefetch_cost_model/build \
  -DMLIR_DIR=/Users/alpaca/Documents/SME/external/llvm-project/build/lib/cmake/mlir \
  -DLLVM_DIR=/Users/alpaca/Documents/SME/external/llvm-project/build/lib/cmake/llvm

cmake --build 02_prefetch_cost_model/build
```

运行分析：

```bash
/Users/alpaca/Documents/SME/external/llvm-project/build/bin/mlir-opt \
  --load-pass-plugin=02_prefetch_cost_model/build/SMEPrefetchCostModelPass.dylib \
  01_custom_lifter/output/gemm_fp32_linalg.mlir \
  --pass-pipeline='builtin.module(convert-linalg-to-loops,canonicalize,func.func(step2-normalize-scf-to-affine,affine-raise-from-memref,canonicalize,step2-prefetch-cost-model))' \
  -o 02_prefetch_cost_model/output/01_affine_analysis.mlir
```

输出：

- [02_prefetch_cost_model/output/01_affine_analysis.mlir](/Users/alpaca/Documents/SME/SME1/02_prefetch_cost_model/output/01_affine_analysis.mlir)
- [02_prefetch_cost_model/output/prefetch_analysis.json](/Users/alpaca/Documents/SME/SME1/02_prefetch_cost_model/output/prefetch_analysis.json)
- [02_prefetch_cost_model/output/prefetch_analysis.md](/Users/alpaca/Documents/SME/SME1/02_prefetch_cost_model/output/prefetch_analysis.md)

这一层不插入预取，只生成决策。当前默认策略是预取 A/B 读流，不主动预取 C。

## 5. 第 03 步：vector 层预取注入

构建 pass：

```bash
cmake -S 03_prefetch_injection \
  -B 03_prefetch_injection/build \
  -DMLIR_DIR=/Users/alpaca/Documents/SME/external/llvm-project/build/lib/cmake/mlir \
  -DLLVM_DIR=/Users/alpaca/Documents/SME/external/llvm-project/build/lib/cmake/llvm

cmake --build 03_prefetch_injection/build
```

先把 linalg 主线转成 vector 层：

```bash
/Users/alpaca/Documents/SME/external/llvm-project/build/bin/mlir-opt \
  01_custom_lifter/output/gemm_fp32_linalg.mlir \
  --pass-pipeline='builtin.module(transform-preload-library{transform-library-paths=03_prefetch_injection/input/linalg_to_vector_transform.mlir},transform-interpreter)' \
  -o 03_prefetch_injection/output/01_vector.mlir
```

再根据第二步决策插入预取：

```bash
/Users/alpaca/Documents/SME/external/llvm-project/build/bin/mlir-opt \
  --load-pass-plugin=03_prefetch_injection/build/SMEPrefetchInjectionPass.dylib \
  03_prefetch_injection/output/01_vector.mlir \
  --pass-pipeline='builtin.module(func.func(step3-inject-vector-prefetch))' \
  -o 03_prefetch_injection/output/02_vector_prefetch.mlir
```

输出：

- [03_prefetch_injection/output/01_vector.mlir](/Users/alpaca/Documents/SME/SME1/03_prefetch_injection/output/01_vector.mlir)
- [03_prefetch_injection/output/02_vector_prefetch.mlir](/Users/alpaca/Documents/SME/SME1/03_prefetch_injection/output/02_vector_prefetch.mlir)

验证重点：

```mlir
memref.prefetch ...
vector.transfer_read ...
vector.outerproduct ...
```

当前第三步会在 A/B 的 rank-1 `vector.transfer_read` 前插入两条静态 `memref.prefetch`。它们位于循环控制流内部，因此运行时会随循环反复执行。

## 6. 第 04 步：vector 预取结果降到 Arm SME / LLVM

构建 pass：

```bash
cmake -S 04_vector_arm_sme_llvm \
  -B 04_vector_arm_sme_llvm/build \
  -DMLIR_DIR=/Users/alpaca/Documents/SME/external/llvm-project/build/lib/cmake/mlir \
  -DLLVM_DIR=/Users/alpaca/Documents/SME/external/llvm-project/build/lib/cmake/llvm

cmake --build 04_vector_arm_sme_llvm/build
```

生成 Arm SME 混合层：

```bash
/Users/alpaca/Documents/SME/external/llvm-project/build/bin/mlir-opt \
  03_prefetch_injection/output/02_vector_prefetch.mlir \
  -test-lower-to-arm-sme \
  -o 04_vector_arm_sme_llvm/output/01_arm_sme_prefetch.mlir
```

生成 LLVM dialect：

```bash
/Users/alpaca/Documents/SME/external/llvm-project/build/bin/mlir-opt \
  03_prefetch_injection/output/02_vector_prefetch.mlir \
  -test-lower-to-arm-sme -test-lower-to-llvm \
  -o /private/tmp/step4_llvm_prefetch_pre_legalize.mlir

/Users/alpaca/Documents/SME/external/llvm-project/build/bin/mlir-opt \
  /private/tmp/step4_llvm_prefetch_pre_legalize.mlir \
  -convert-vector-to-llvm=enable-arm-sve -reconcile-unrealized-casts \
  -o /private/tmp/step4_llvm_prefetch_legalized.mlir

/Users/alpaca/Documents/SME/external/llvm-project/build/bin/mlir-opt \
  --load-pass-plugin=04_vector_arm_sme_llvm/build/SMEStep4LoweringPasses.dylib \
  --pass-pipeline='builtin.module(step4-repair-llvm-index-bridges)' \
  /private/tmp/step4_llvm_prefetch_legalized.mlir \
  -o 04_vector_arm_sme_llvm/output/02_llvm_prefetch.mlir
```

输出：

- [04_vector_arm_sme_llvm/output/01_arm_sme_prefetch.mlir](/Users/alpaca/Documents/SME/SME1/04_vector_arm_sme_llvm/output/01_arm_sme_prefetch.mlir)
- [04_vector_arm_sme_llvm/output/02_llvm_prefetch.mlir](/Users/alpaca/Documents/SME/SME1/04_vector_arm_sme_llvm/output/02_llvm_prefetch.mlir)

验证重点：

- `01_arm_sme_prefetch.mlir` 中应能看到 `memref.prefetch` 和 `arm_sme.intr.*`
- `02_llvm_prefetch.mlir` 中应能看到 `llvm.intr.prefetch`

注意：`-test-lower-to-arm-sme` 是 MLIR 工具链自带的测试 lowering pipeline，不是本仓库编写的 pass。本仓库第 04 步自定义 pass 只负责 `step4-repair-llvm-index-bridges`。

## 7. 第 05 步：生成 LLVM IR、汇编和可执行文件

执行：

```bash
python3 05_native_build_run/build_and_run.py
```

该脚本会依次执行：

```text
mlir-translate --mlir-to-llvmir
llc -mattr=+sme -filetype=asm
llc -mattr=+sme -filetype=obj
clang step4_llvm_prefetch.o native_harness.c
```

输出：

- [05_native_build_run/output/step4_llvm_prefetch.ll](/Users/alpaca/Documents/SME/SME1/05_native_build_run/output/step4_llvm_prefetch.ll)
- [05_native_build_run/output/step4_llvm_prefetch.s](/Users/alpaca/Documents/SME/SME1/05_native_build_run/output/step4_llvm_prefetch.s)
- `05_native_build_run/output/step4_llvm_prefetch.o`
- `05_native_build_run/output/step4_llvm_prefetch_demo`
- [05_native_build_run/output/step4_llvm_prefetch_demo.log](/Users/alpaca/Documents/SME/SME1/05_native_build_run/output/step4_llvm_prefetch_demo.log)
- [05_native_build_run/output/summary.txt](/Users/alpaca/Documents/SME/SME1/05_native_build_run/output/summary.txt)

验证重点：

```bash
rg -n "llvm.prefetch|prfm|fmopa|smstart" \
  05_native_build_run/output/step4_llvm_prefetch.ll \
  05_native_build_run/output/step4_llvm_prefetch.s
```

含义：

- `llvm.prefetch` 表示 LLVM IR 中保留了预取 intrinsic
- `prfm` 表示汇编中出现 AArch64 预取指令
- `fmopa` 表示汇编中出现 SME matrix outer product 指令
- `smstart` 表示汇编中进入 SME streaming / ZA 状态

  05_native_build_run/output/step4_llvm_prefetch.ll \

```text

unified_mainline_object: ok
unified_mainline_link: ok
unified_mainline_run: ok
```

如果这些项均为 `ok`，说明当前仓库已经完成从 C kernel 到可执行文件的端到端流程。
