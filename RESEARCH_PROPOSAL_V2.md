# 面向 ARM SME 的高层 IR 驱动预取优化研究方案（重组版）

## 一句话概述

针对 ARM SME 场景下传统编译流程难以保留矩阵计算高层语义、现有预取优化主要依赖后端启发式规则、难以结合循环分块与数据复用特征做精细决策的问题，本研究拟采用“受限 C kernel + 自定义提升器 + 高层 MLIR 语义分析 + 预取语义注入 + ArmSME/LLVM lowering”的方法，实现一套面向 GEMM/FlagGEMM 的高层 IR 驱动预取优化链路，并期望在 LX2 实机上获得优于无预取 baseline 和毕昇原生预取的性能结果。

---

## 1. 问题

本研究聚焦的问题可以概括为以下三类。

### 1.1 高层计算语义在传统编译链中快速丢失

对于面向 ARM SME 的矩阵计算程序，若直接采用传统的：

```text
C/C++ -> Clang -> LLVM IR
```

编译流程，则源程序中的很多高层信息会很快退化为低层指针、load/store 和标量算术，难以继续保留：

- 矩阵乘的整体算子语义
- 归约维与空间维信息
- `mc / nc / kc / mr / nr` 分块结构
- 局部子块与数据复用关系
- 适合做预取决策的访存模式信息

这使得后续很难在编译中端阶段针对 SME 场景进行结构化优化。

### 1.2 现有预取优化主要依赖后端启发式，缺少高层语义支撑

现有预取优化通常出现在 LLVM 或目标后端，主要依赖：

- 局部访存模式
- 静态启发式规则
- 硬件自动预取器的假设

这种方式的问题在于：

- 难以直接感知矩阵乘中的块结构与复用层次
- 难以区分“应保留缓存”与“应流式处理”的数据流
- 难以对 `KEEP / STRM`、预取距离、缓存层级进行统一建模

因此，对于 SME 场景下的大规模分块矩阵计算，单纯依赖后端启发式未必能得到最优效果。

### 1.3 预取语义在跨层 lowering 时容易丢失

即使在高层 IR 中做出了合理的预取决策，仍然存在一个关键难点：

- 如何将预取语义从 `linalg / affine / scf / vector`
- 一直保留到 `arm_sme` / LLVM IR
- 并最终映射成毕昇能够识别的目标预取形式

其中需要解决的问题包括：

- 预取距离如何表达
- `KEEP / STRM` 如何表达
- 是否需要目标相关 intrinsic
- 语义在 lowering 过程中是否丢失

## 2. 分析

基于当前问题，可以把整个研究拆解成“表示问题、决策问题、落地问题”三层。

### 2.1 表示问题：必须先把 C kernel 恢复成高层 MLIR 结构

要研究 SME 相关的高层 IR 形式，不能把 `clang -> LLVM IR` 当作主路线，而必须先构建：

```text
受限 C kernel -> 高层 MLIR
```

这里的核心分析结论是：

- `C = A * B` 的数学语义应优先映射成 `linalg.matmul`
- 外层 `i / j / k` 块循环适合映射成 `scf.for` 或 `affine.for`
- 内层 `ii / jj / kk` 微块结构可继续作为 tile / micro-kernel 结构保留

也就是说，研究起点不应是 LLVM IR，而应是“可恢复算子语义和循环结构的高层 MLIR”。

### 2.2 决策问题：预取策略必须建立在高层语义分析上

在高层 MLIR 上，能够更自然地分析：

- 数据流是小而热还是大而冷
- 数据是否存在短期重用
- 访问是否规则、顺序、跨 cache line
- 当前块的工作集是否适合保留在缓存中

进而生成如下类型的决策：

- 是否预取
- 预取读还是写
- 预取距离 `D`
- 目标缓存层级
- `KEEP / STRM`

因此，高层语义分析不是附属步骤，而是整个预取优化方案的核心。

### 2.3 落地问题：需要一条可控的跨层 lowering 链路

从研究落地角度看，必须建立一条明确的跨层路径：

```text
linalg / scf / affine
-> vector
-> arm_sme
-> llvm
-> 毕昇
-> LX2
```

这样才能回答最终的工程问题：

- 预取语义是否被保留下来
- 目标 intrinsic 是否被识别
- 汇编中是否真的出现目标预取指令
- 性能是否优于 baseline

## 3. 方法

针对上述问题，本研究拟采用以下方法。

### 3.1 采用受限 C kernel 作为统一输入

输入程序使用规则分块 GEMM kernel，其特点包括：

- 仅保留 `C = A * B`
- 保留 `mc / nc / kc / mr / nr`
- 使用规则循环
- 使用规则矩阵访存
- 不引入显式 SME intrinsic 和复杂运行时机制

这样做的目的，是把输入程序控制在“可自动识别的子语言”范围内，降低提升器实现难度。

### 3.2 采用自定义提升器将 C 程序转成高层 MLIR

自定义提升器负责将 C 程序恢复成高层 MLIR 结构，而不是依赖 `import-llvm`。

目标输出包括：

- `linalg.matmul`
- `scf.for`
- `affine.for`

这样就能显式保留：

- 矩阵乘算子语义
- tile 结构
- 归约维 / 空间维
- 局部输出子块 `Cr`

### 3.3 采用高层语义分析模型生成预取策略

在 `linalg` 与 `affine/scf` 层，构建语义分析模块，提取：

- 数据流性质
- 访存模式
- 复用距离
- 工作集特征

在此基础上构建 Cost Model，生成：

- 预取目标
- 预取距离
- `KEEP / STRM`
- 缓存层级策略

### 3.4 采用自定义预取语义表示和 MLIR 注入机制

为了避免预取语义在 lowering 中丢失，本研究拟采用：

- 自定义预取 op
- 或统一 attribute 规范

在高层 IR 中表达：

- 读 / 写预取
- 预取距离
- `KEEP / STRM`
- 目标缓存层级

然后通过：

- 自定义 MLIR pass
- 或 Transform Dialect 脚本

将其注入到 IR 中。

### 3.5 采用 LLVM/MLIR 现有链路完成 vector 与 ArmSME lowering

在预取注入后，继续使用本地 LLVM/MLIR 已有路径：

```text
linalg -> vector -> arm_sme -> llvm
```

重点观察：

- `vector.contract`
- `vector.outerproduct`
- `arm_sme.outerproduct`
- `arm_sme.fmopa`

及其与预取语义的协同。

## 4. 实现

整个方案的实现分为五个模块。

### 4.1 模块一：输入程序与语义标注

输入程序采用当前仓库中的 [gemm_mlir_kernel.c](/Users/alpaca/Documents/SME/SME1/gemm_mlir_kernel.c) 作为 V1 基线。

其中通过注释显式提供：

- `@kernel`
- `@semantic`
- `@layout`
- `@blocking`
- `@lift-target`

这些信息将作为提升器的辅助输入。

### 4.2 模块二：C -> 高层 MLIR 提升器

这是整个系统的入口模块，主要负责：

- 识别外层 `i / j / k` 分块循环
- 识别内层 `ii / jj / kk` 微块与归约循环
- 识别 `Cr += A * B` 乘加模式
- 构造高层 MLIR 结构

建议实现方式：

- 原型阶段：`clang -Xclang -ast-dump=json` + 脚本
- 正式阶段：Clang AST / LibTooling + MLIR API

### 4.3 模块三：高层语义分析与 Cost Model

该模块在 `linalg` / `affine` / `scf` 层工作，负责：

- 分析矩阵块的重用关系
- 分析访存流是热数据还是冷数据
- 分析工作集大小
- 推导预取距离 `D`
- 给出 `KEEP / STRM` 等策略建议

输出是一份“预取决策结果”，供后续注入模块使用。

### 4.4 模块四：预取语义注入与 lowering pass

该模块负责：

- 在高层 IR 中插入预取语义
- 设计或使用统一的预取表示
- 保证语义可传递到更低层

这一部分至少需要实现：

- 一个最小版预取 op / attribute 设计
- 一个对应的注入 pass
- 一个 lowering pass，把该语义传到 LLVM IR

### 4.5 模块五：LLVM / 毕昇 / LX2 验证链路

该模块负责：

- 检查 LLVM IR 中的预取语义
- 检查毕昇能否接受
- 检查汇编中是否出现目标预取形式
- 在 LX2 上做功能和性能验证

## 5. 结果

本研究预期得到的结果分为“功能结果”和“性能结果”两类。

### 5.1 功能结果

预期实现以下功能性成果：

1. 能够把受限 C GEMM kernel 提升为高层 MLIR
2. 能够在高层 MLIR 中表示矩阵乘与循环分块语义
3. 能够基于高层语义自动生成预取策略
4. 能够把预取语义注入到 MLIR 中
5. 能够把该语义继续 lowering 到 LLVM IR
6. 能够被毕昇编译器接受并生成目标代码

### 5.2 性能结果

预期在 LX2 上获得以下结果：

1. 相比 **Baseline 1 (No Prefetch)**，L2 miss 率下降
2. 相比 **Baseline 1**，GFLOPS 提升
3. 在规则 dense GEMM 上，性能优于仅依赖硬件自动预取的方案
4. 在部分场景下，有机会优于 **Baseline 2 (BiSheng Native Prefetch)**

这里的核心结果目标可以概括为：

- 预取得到可观测的功能正确性结果
- 预取得到可量化的性能收益结果

### 5.3 V1 与扩展结果

V1 阶段预期只覆盖：

- FP32 dense GEMM
- 规则 unit-stride 读预取
- 基于高层语义的最小可用预取策略

在 V1 跑通后，再扩展到：

- 不规则控制流
- FlagGEMM
- 稀疏度变化
- CPU / SVE / SME 并行调度场景

---

## 6. 当前最小落地计划

为了尽快得到第一批结果，建议按以下顺序推进：

1. 固定输入 kernel 形式
2. 手工写一个对应的 `linalg.matmul` 版 MLIR
3. 跑通 `linalg -> vector -> arm_sme -> llvm` 现有链路
4. 实现最小版 `C -> 高层 MLIR` 提升器
5. 定义最小版预取 op / attribute
6. 实现预取注入 pass
7. 生成 LLVM IR 并接毕昇
8. 在 LX2 上完成 baseline 与 proposed 对比实验

---

## 7. 结论

本研究的核心不是单纯讨论 SME 指令本身，而是围绕以下主线展开：

- 针对高层语义在传统编译链中快速丢失、后端预取缺乏高层结构支撑的问题
- 采用受限 C kernel、自定义提升器、高层 MLIR 语义分析与自定义预取语义注入的方法
- 实现一条从高层 MLIR 到 ArmSME/LLVM 再到毕昇/LX2 的完整实验链路
- 期望得到优于 baseline 的预取效果与最终性能收益

这也是“探索面向 ARM SME 的高层 IR 驱动预取优化方法”的具体落地形式。
