# 04 输出文件内容说明

本文档说明第 04 步如何接收第三步生成的 vector 层预取 MLIR，并继续产生 Arm SME 层和 LLVM dialect 层输出。

当前主线是：

```text
03_prefetch_injection/output/02_vector_prefetch.mlir
-> 04_vector_arm_sme_llvm/output/01_arm_sme_prefetch.mlir
-> 04_vector_arm_sme_llvm/output/02_llvm_prefetch.mlir
```

需要说明的是，`01_arm_sme_prefetch.mlir` 和 `02_llvm_prefetch.mlir` 是自动 lowering 后的机器化 IR，包含大量重复展开的 basic block、tile slice、load/store、predicate mask 和 SME intrinsic。本文按“语句类别 + 代表片段”解释每类语句的作用。

## 1. 第四步输入：第三步 vector 预取 MLIR

对应文件：

- [../03_prefetch_injection/output/02_vector_prefetch.mlir](/Users/alpaca/Documents/SME/SME1/03_prefetch_injection/output/02_vector_prefetch.mlir)

这一文件已经完成两件事：

- 矩阵乘从 `linalg.matmul` 变成 `vector.transfer_read`、`vector.outerproduct`、`vector.transfer_write`。
- 第三步根据第二步分析结果，在 A/B 的 rank-1 `vector.transfer_read` 前插入 `memref.prefetch`。

典型结构如下：

```mlir
memref.prefetch %subview_10[%c0], read, locality<3>, data
%14 = vector.transfer_read %subview_10[%c0], ...

memref.prefetch %subview_11[%c0], read, locality<3>, data
%15 = vector.transfer_read %subview_11[%c0], ...

%17 = vector.mask %13 {
  vector.outerproduct %14, %15, %16 {kind = #vector.kind<add>}
}
vector.transfer_write %17, ...
```

含义是：

- 第一组 `memref.prefetch + vector.transfer_read` 对应 A 的向量读流。
- 第二组 `memref.prefetch + vector.transfer_read` 对应 B 的向量读流。
- `vector.outerproduct` 表示 A 向量和 B 向量做外积，并累加到 C tile。
- `vector.transfer_write` 把更新后的 C tile 写回内存。

第 04 步从这里开始，不再重新生成或修改这一层。

## 2. `01_arm_sme_prefetch.mlir`

对应文件：

- [output/01_arm_sme_prefetch.mlir](/Users/alpaca/Documents/SME/SME1/04_vector_arm_sme_llvm/output/01_arm_sme_prefetch.mlir)

这个文件由第三步输出经过：

```bash
mlir-opt 03_prefetch_injection/output/02_vector_prefetch.mlir -test-lower-to-arm-sme
```

生成。它的作用是观察 vector 计算主线如何进入 Arm SME 相关表示。

### 2.1 函数属性

```mlir
attributes {llvm.arm_locally_streaming, llvm.arm_new_za, step3.prefetch_injected = "vector.memref.prefetch"}
```

- `llvm.arm_locally_streaming` 表示函数需要 Arm streaming mode。
- `llvm.arm_new_za` 表示函数需要新的 ZA tile 状态。
- `step3.prefetch_injected = "vector.memref.prefetch"` 来自第三步，说明当前预取不是第四步重新插入的。

### 2.2 SME tile 临时内存

```mlir
%alloca = memref.alloca(%c4_vscale, %c4_vscale)
  {arm_sme.in_memory_tile_id = 27 : i32} : memref<?x?xf32>
```

- `memref.alloca` 在栈上分配临时 tile 存储。
- `%c4_vscale` 表示 `4 * vscale`，是 SME lowering 中使用的 tile slice 大小。
- `arm_sme.in_memory_tile_id` 标记这个内存 tile 与 SME tile id 的关系。
- 这类语句会重复出现，因为 16x16 scalable tile 会被拆成多个 4x4 scalable tile 片段。

### 2.3 控制流从 `scf.for` 变成 `cf.br`

```mlir
cf.br ^bb1(%c0 : index)
^bb1(%4: index):
  %5 = arith.cmpi slt, %4, %dim : index
  cf.cond_br %5, ^bb2, ^bb70
```

- `cf.br` 是 control flow dialect 的无条件跳转。
- `^bb1(%4: index)` 是 basic block，`%4` 是 block 参数。
- `arith.cmpi slt` 比较当前循环变量是否小于上界。
- `cf.cond_br` 根据比较结果跳到循环体或退出块。
- 这类结构是 `scf.for` 被降级后的显式控制流形式。

### 2.4 预取仍保留为 `memref.prefetch`

```mlir
memref.prefetch %subview_20[%c0], read, locality<3>, data
memref.prefetch %subview_21[%c0], read, locality<3>, data
```

- 第一条预取对应 A 的一维读流。
- 第二条预取对应 B 的一维读流。
- 进入 Arm SME 相关层后，预取仍没有丢失。
- 此时还没有变成 `llvm.intr.prefetch`，因为文件尚未完全降到 LLVM dialect。

### 2.5 masked load 和 vector insert

```mlir
%19 = memref.load %subview_20[%15] : memref<?xf32, ...>
%20 = vector.insert %19, %16 [%15] : f32 into vector<[16]xf32>
```

- `memref.load` 从 A 或 B 的一维 view 中读取一个标量。
- `vector.insert` 把标量插入 vector 的某个 lane。
- 这说明部分 `vector.transfer_read` 已被展开成显式循环、条件判断、标量 load 和 vector insert。

### 2.6 SME intrinsic

```mlir
"arm_sme.intr.read.horiz"(...)
"arm_sme.intr.ld1w.horiz"(...)
"arm_sme.intr.mopa"(...)
"arm_sme.intr.st1w.horiz"(...)
```

- `arm_sme.intr.read.horiz` 从 ZA tile 中读取水平 slice。
- `arm_sme.intr.ld1w.horiz` 把内存中的 `f32` 数据加载到 ZA tile 的水平 slice。
- `arm_sme.intr.mopa` 执行 SME matrix outer product accumulate，是矩阵乘核心。
- `arm_sme.intr.st1w.horiz` 把 ZA tile 的水平 slice 存回内存。

### 2.7 为什么还有 `unrealized_conversion_cast`

```mlir
builtin.unrealized_conversion_cast ...
```

- 这是 MLIR lowering 过程中暂时保留的类型桥接。
- 在 `01_arm_sme_prefetch.mlir` 中出现是正常的，因为它不是最终 LLVM dialect 文件。
- 后续 `-test-lower-to-llvm`、`-reconcile-unrealized-casts` 和自定义 repair pass 会继续清理这些桥接。

## 3. `02_llvm_prefetch.mlir`

对应文件：

- [output/02_llvm_prefetch.mlir](/Users/alpaca/Documents/SME/SME1/04_vector_arm_sme_llvm/output/02_llvm_prefetch.mlir)

这是第 04 步最终输出。它已经降到 LLVM dialect，并且可以用 `mlir-translate --mlir-to-llvmir` 转成 LLVM IR `.ll`。

### 3.1 LLVM dialect 函数签名

```mlir
llvm.func @gemm_fp32_linalg(
  %arg0: !llvm.ptr, %arg1: !llvm.ptr, %arg2: i64, ...
)
attributes {arm_locally_streaming, arm_new_za, step3.prefetch_injected = "vector.memref.prefetch"}
```

- `llvm.func` 表示函数已经进入 LLVM dialect。
- `!llvm.ptr` 是 LLVM 指针类型。
- 多个 `i64` 参数来自 memref descriptor 的 offset、size、stride 等字段。
- `arm_locally_streaming` 和 `arm_new_za` 继续保留 SME 函数属性。
- `step3.prefetch_injected` 说明最终 LLVM 结果仍可追踪到第三步预取注入。

### 3.2 常量、poison 和 descriptor 组装

```mlir
%0 = llvm.mlir.constant(0 : index) : i64
%10 = llvm.mlir.poison : vector<[4]xf32>
%17 = llvm.mlir.poison : !llvm.struct<...>
%18 = llvm.insertvalue %arg14, %17[0] : !llvm.struct<...>
```

- `llvm.mlir.constant` 生成 LLVM dialect 中的常量。
- `llvm.mlir.poison` 生成未定义初值，通常作为 vector 或 struct 的构造起点。
- `llvm.insertvalue` 把参数逐个填入 struct 字段，用来重建 memref descriptor。

### 3.3 vscale 与栈上 tile 缓冲区

```mlir
%25 = "llvm.intr.vscale"() : () -> i64
%26 = llvm.mul %25, %12 : i64
%28 = llvm.alloca %27 x f32 : (i64) -> !llvm.ptr
```

- `llvm.intr.vscale` 读取目标机器的 scalable vector 缩放因子。
- `llvm.mul` 计算运行时 tile 大小。
- `llvm.alloca` 分配 LLVM 层栈内存，用于保存 lowering 后的 tile 缓冲。

### 3.4 LLVM basic block 和分支

```mlir
llvm.br ^bb1(%0 : i64)
^bb1(%52: i64):
  %53 = llvm.icmp "slt" %52, %arg3 : i64
  llvm.cond_br %53, ^bb2, ^bb271
```

- `llvm.br` 是 LLVM dialect 的无条件跳转。
- `^bb1(%52: i64)` 是带参数的 basic block。
- `llvm.icmp "slt"` 是有符号小于比较。
- `llvm.cond_br` 是条件跳转。
- 这类语句共同表示 lowering 后的显式循环控制流。

### 3.5 mask 构造

```mlir
%56 = llvm.intr.stepvector : vector<[16]xi32>
%58 = llvm.trunc %57 : i64 to i32
%59 = llvm.insertelement %58, %14[%15 : i32] : vector<[16]xi32>
%60 = llvm.shufflevector %59, %14 [...]
%61 = llvm.icmp "slt" %56, %60 : vector<[16]xi32>
```

- `llvm.intr.stepvector` 生成 lane 编号向量，例如 `[0, 1, 2, ...]`。
- `llvm.trunc` 把 `i64` 长度截断成 `i32`，供向量 mask 构造使用。
- `llvm.insertelement` 把长度值插入向量第 0 个 lane。
- `llvm.shufflevector` 把该长度值广播到所有 lane。
- `llvm.icmp "slt"` 比较 lane id 是否小于有效长度，生成 mask。

### 3.6 最终预取语句

```mlir
%84 = llvm.getelementptr %82[%83] : (!llvm.ptr, i64) -> !llvm.ptr, f32
"llvm.intr.prefetch"(%84) <{cache = 1 : i32, hint = 3 : i32, rw = 0 : i32}> : (!llvm.ptr) -> ()
```

- `llvm.getelementptr` 计算预取地址。
- `llvm.intr.prefetch` 是 LLVM dialect 的预取 intrinsic。
- `rw = 0` 表示读预取。
- `hint = 3` 表示高局部性。
- `cache = 1` 表示数据 cache。
- 这条预取来自第三步 `memref.prefetch`。

### 3.7 load、insert 和向量构造

```mlir
%92 = llvm.load %91 : !llvm.ptr -> f32
%93 = llvm.insertelement %92, %86[%85 : i64] : vector<[16]xf32>
```

- `llvm.load` 从计算出的 LLVM 指针地址中读取一个 `f32`。
- `llvm.insertelement` 把标量读入结果插入向量 lane。
- 这是 vector transfer lowering 后的显式 LLVM dialect 表示。

### 3.8 SME/SVE intrinsic

```mlir
"arm_sme.intr.ld1w.horiz"(...)
"arm_sme.intr.mopa"(...)
"arm_sme.intr.st1w.horiz"(...)
"arm_sve.intr.psel"(...)
```

- `arm_sme.intr.ld1w.horiz` 对应 SME 水平加载。
- `arm_sme.intr.mopa` 对应 SME 矩阵外积累加，是计算核心。
- `arm_sme.intr.st1w.horiz` 对应 SME 水平存储。
- `arm_sve.intr.psel` 用于处理 predicate select，配合 mask 控制有效 lane。

### 3.9 为什么最终没有 `memref.prefetch`

`02_llvm_prefetch.mlir` 中不再出现 `memref.prefetch`，因为它已经被 lowering 成：

```mlir
"llvm.intr.prefetch"(...)
```

这说明预取语义完成了从 MLIR memref 层到 LLVM dialect 层的转换。

## 4. 输出关系总结

| 文件 | 主要层级 | 预取形式 | 计算形式 | 作用 |
| --- | --- | --- | --- | --- |
| `../03_prefetch_injection/output/02_vector_prefetch.mlir` | vector / memref / scf | `memref.prefetch` | `vector.outerproduct` | 第四步输入，证明第三步已完成 vector 预取注入 |
| `output/01_arm_sme_prefetch.mlir` | Arm SME 混合层 | `memref.prefetch` | `arm_sme.intr.*` 开始出现 | 证明带预取主线可以进入 Arm SME |
| `output/02_llvm_prefetch.mlir` | LLVM dialect | `llvm.intr.prefetch` | `arm_sme.intr.*` / `llvm.*` | 证明最终可导出 LLVM IR，且预取未丢失 |

## 5. 阅读这些文件时应重点观察什么

建议按下面顺序看：

1. 在第三步输出中找 `memref.prefetch`，确认它位于 A/B 的 `vector.transfer_read` 前。
2. 在第三步输出中找 `vector.outerproduct`，确认矩阵乘已经以向量外积表达。
3. 在 `01_arm_sme_prefetch.mlir` 中找 `arm_sme.intr.mopa`，确认计算开始进入 SME 语义。
4. 在 `02_llvm_prefetch.mlir` 中找 `llvm.intr.prefetch`，确认预取进入 LLVM dialect。
5. 在 `02_llvm_prefetch.mlir` 中确认没有 `unrealized_conversion_cast` 或 `arith.index_cast` 残留，说明最终文件已被 repair pass 收口。

一句话总结：第 04 步两个输出文件展示了第三步 vector 预取主线继续降低到 Arm SME 表达，再到 LLVM dialect 预取 intrinsic 的完整过程。
