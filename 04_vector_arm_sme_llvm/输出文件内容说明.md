# 04 输出文件内容说明

本文档说明第 04 步 `output/` 目录中三个主线 MLIR 文件的内容，重点回答：

```text
01_vector_prefetch.mlir
```

需要说明的是，`02_arm_sme_prefetch.mlir` 和 `03_llvm_prefetch.mlir` 是 MLIR 自动 lowering 后的机器化 IR，分别约一千多行和两千多行。它们包含大量重复展开的 basic block、tile slice、load/store、predicate mask 和 SME intrinsic。本文不会逐行复制整份文件，而是按“语句类别 + 代表片段”解释每类语句的作用；这些类别覆盖了文件中实际出现的语句结构。

## 1. `01_vector_prefetch.mlir`

对应文件：

- [output/01_vector_prefetch.mlir](/Users/alpaca/Documents/SME/SME1/04_vector_arm_sme_llvm/output/01_vector_prefetch.mlir)

这是第 04 步最重要的高层主线输出。它已经从 `linalg.matmul` 降到 vector 结构，并且第 04 步 pass 已经在 A/B 的向量读前插入 `memref.prefetch`。

### 1.1 文件头部 affine map

```mlir
#map = affine_map<(d0)[s0, s1] -> (-d0 + s0, s1)>
#map1 = affine_map<(d0)[s0, s1] -> (d0 * s1 + s0)>
```

- `#map` 用来计算当前 tile 的有效大小。`-d0 + s0` 表示剩余长度，`s1` 表示最大 tile 长度，后面配合 `affine.min` 取二者较小值。
- `#map1` 是 rank-1 subview 的线性化地址映射，用于把一维下标映射到实际内存偏移。

### 1.2 module 与函数签名

```mlir
module {
  func.func @gemm_step4_compute(
    %arg0: memref<?x?xf32, strided<[?, ?], offset: ?>>,
    %arg1: memref<?x?xf32, strided<[?, ?], offset: ?>>,
    %arg2: memref<?x?xf32, strided<[?, ?], offset: ?>>
  ) -> memref<?x?xf32, strided<[?, ?], offset: ?>>
  attributes {step4_vector_prefetch = "memref_prefetch"} {
```

- `module` 是 MLIR 顶层容器。
- `func.func @gemm_step4_compute` 是当前矩阵乘 kernel。
- `%arg0` 表示矩阵 `A`。
- `%arg1` 表示矩阵 `B`。
- `%arg2` 表示矩阵 `C`，也是输出缓冲区。
- `memref<?x?xf32>` 表示二维动态大小的 `float` 矩阵。
- `strided<[?, ?], offset: ?>` 表示矩阵布局的 stride 和 offset 在运行时确定。
- `step4_vector_prefetch = "memref_prefetch"` 是第 04 步 pass 添加的标记，说明此函数已经插入 vector 层预取。

### 1.3 常量和矩阵维度

```mlir
%0 = ub.poison : f32
%c16 = arith.constant 16 : index
%c1 = arith.constant 1 : index
%c0 = arith.constant 0 : index
%dim = memref.dim %arg0, %c0
%dim_0 = memref.dim %arg0, %c1
%dim_1 = memref.dim %arg1, %c1
```

- `%0 = ub.poison : f32` 创建一个未定义的 `f32` 填充值，供 masked vector read 在越界 lane 上使用。
- `%c16` 是 tile 的基础向量长度。
- `%c1` 是归约维 `K` 的步长。
- `%c0` 是零下标。
- `%dim` 读取 `A` 的第 0 维，也就是 `M`。
- `%dim_0` 读取 `A` 的第 1 维，也就是 `K`。
- `%dim_1` 读取 `B` 的第 1 维，也就是 `N`。

### 1.4 scalable vector 长度

```mlir
%vscale = vector.vscale
%c16_vscale = arith.muli %vscale, %c16 : index
%vscale_2 = vector.vscale
%c16_vscale_3 = arith.muli %vscale_2, %c16 : index
```

- `vector.vscale` 表示目标机器的 scalable vector 缩放因子。
- `%c16_vscale` 表示 `16 * vscale`，是 M 方向 tile 步长。
- `%c16_vscale_3` 表示 `16 * vscale`，是 N 方向 tile 步长。
- 这里使用 scalable vector，是为了适配 SVE/SME 这类向量长度可伸缩的架构。

### 1.5 外层 tile 循环

```mlir
scf.for %arg3 = %c0 to %dim step %c16_vscale {
  %1 = affine.min #map(%arg3)[%dim, %c16_vscale]
  %2 = vector.create_mask %1 : vector<[16]xi1>
```

- `scf.for %arg3` 是 M 方向 tile 循环。
- 循环从 0 到 `M`，每次推进 `16 * vscale`。
- `%1 = affine.min ...` 计算当前 M tile 的实际有效长度，处理尾块。
- `%2 = vector.create_mask %1` 创建 M 方向的一维 mask，保证尾块不会越界读取。

```mlir
scf.for %arg4 = %c0 to %dim_1 step %c16_vscale_3 {
  %3 = affine.min #map(%arg4)[%dim_1, %c16_vscale_3]
```

- `scf.for %arg4` 是 N 方向 tile 循环。
- `%3` 是当前 N tile 的实际有效长度。

### 1.6 C tile 和 mask

```mlir
%subview = memref.subview %arg2[%arg3, %arg4] [%1, %3] [1, 1]
%4 = vector.create_mask %3 : vector<[16]xi1>
%5 = vector.create_mask %1, %3 : vector<[16]x[16]xi1>
%6 = vector.create_mask %1, %3 : vector<[16]x[16]xi1>
%subview_4 = memref.subview %arg2[%arg3, %arg4] [%1, %3] [1, 1]
```

- `%subview` 取出当前要计算的 `C[i:i+tileM, j:j+tileN]` tile。
- `%4` 创建 N 方向一维 mask。
- `%5` 创建二维 C tile mask，用于读取/写回 C。
- `%6` 创建二维 outerproduct mask，用于控制有效计算 lane。
- `%subview_4` 是同一块 C tile 的另一个 view，后面用于 `memref.copy`。

### 1.7 K 方向归约循环

```mlir
scf.for %arg5 = %c0 to %dim_0 step %c1 {
```

- `%arg5` 是 K 方向归约变量。
- 当前示例中 K 每次走 1，这是由输入 transform 调度产生的微内核形式。
- 每一轮读取 A 的一列向量和 B 的一行向量，做一次 outer product 累加到 C tile。

### 1.8 A/B 子视图

```mlir
%subview_5 = memref.subview %arg0[%arg3, %arg5] [%1, 1] [1, 1]
%subview_6 = memref.subview %arg1[%arg5, %arg4] [1, %3] [1, 1]
%subview_7 = memref.subview %subview_5[0, 0] [%1, 1] [1, 1]
%subview_8 = memref.subview %subview_6[0, 0] [1, %3] [1, 1]
```

- `%subview_5` 取出 A 的当前列块，即 `A[i:i+tileM, k]`。
- `%subview_6` 取出 B 的当前行块，即 `B[k, j:j+tileN]`。
- `%subview_7` 把 A 的 `?x1` view 简化成一维 view，便于 vector read。
- `%subview_8` 把 B 的 `1x?` view 简化成一维 view，便于 vector read。

### 1.9 第 04 步插入的预取语句

```mlir
memref.prefetch %subview_7[%c0], read, locality<3>, data : memref<?xf32, #map1>
%7 = vector.transfer_read %subview_7[%c0], %0, %2 {in_bounds = [true]}
```

- `memref.prefetch %subview_7[%c0]` 是对 A 向量读的预取。
- `read` 表示读预取。
- `locality<3>` 表示高局部性，希望数据尽量保留在 cache。
- `data` 表示数据 cache 预取，而不是指令 cache 预取。
- `%7 = vector.transfer_read ...` 才是真正执行 A 向量读取。
- 预取语句位于读取之前，表示提前请求即将读取的数据。

```mlir
memref.prefetch %subview_8[%c0], read, locality<3>, data : memref<?xf32, #map1>
%8 = vector.transfer_read %subview_8[%c0], %0, %4 {in_bounds = [true]}
```

- 这两句对应 B 的预取和向量读取。
- `%8` 是 B 的一维向量。

### 1.10 C tile 读取、outer product 和写回

```mlir
%9 = vector.transfer_read %subview[%c0, %c0], %0, %5
%10 = vector.mask %6 {
  vector.outerproduct %7, %8, %9 {kind = #vector.kind<add>}
}
vector.transfer_write %10, %subview[%c0, %c0], %5
```

- `%9` 读取当前 C tile。
- `vector.outerproduct %7, %8, %9` 表示用 A 向量 `%7` 和 B 向量 `%8` 做外积，并累加到 C tile `%9`。
- `{kind = #vector.kind<add>}` 表示 outer product 的结果与累加器相加。
- `vector.mask %6` 让 outer product 只在有效 lane 上执行。
- `vector.transfer_write` 把更新后的 C tile 写回内存。

### 1.11 copy、return 和空 transform module

```mlir
memref.copy %subview, %subview_4
return %arg2 : memref<?x?xf32, strided<[?, ?], offset: ?>>
module attributes {transform.with_named_sequence} {
}
```

- `memref.copy` 是 bufferization/lowering 过程中保留下来的内存同步操作。
- `return %arg2` 返回输出矩阵 C 的 memref。
- 空的 `transform.with_named_sequence` module 表示 transform 调度已经被执行并擦除，只留下空壳。

## 2. `02_arm_sme_prefetch.mlir`

对应文件：

- [output/02_arm_sme_prefetch.mlir](/Users/alpaca/Documents/SME/SME1/04_vector_arm_sme_llvm/output/02_arm_sme_prefetch.mlir)

这个文件由 `01_vector_prefetch.mlir` 经过：

```bash
mlir-opt -test-lower-to-arm-sme
```

生成。它的作用是观察 vector 计算主线如何进入 Arm SME 相关表示。

### 2.1 函数属性

```mlir
attributes {llvm.arm_locally_streaming, llvm.arm_new_za, step4_vector_prefetch = "memref_prefetch"}
```

- `llvm.arm_locally_streaming` 表示函数需要 Arm streaming mode。
- `llvm.arm_new_za` 表示函数需要新的 ZA tile 状态。
- `step4_vector_prefetch = "memref_prefetch"` 说明第 04 步预取插入仍然保留在主线中。

### 2.2 SME tile 临时内存

```mlir
%alloca = memref.alloca(%c4_vscale, %c4_vscale)
  {arm_sme.in_memory_tile_id = 27 : i32} : memref<?x?xf32>
```

- `memref.alloca` 在栈上分配临时 tile 存储。
- `%c4_vscale` 表示 `4 * vscale`，是 SME lowering 中使用的 tile slice 大小。
- `arm_sme.in_memory_tile_id` 标记这个内存 tile 与 SME tile id 的关系。
- 这类语句会重复出现，因为 16x16 scalable tile 会被拆成多个 4x4 scalable tile 片段。

### 2.3 控制流从 `scf.for` 变成 `cf.br`

```mlir
cf.br ^bb1(%c0 : index)
^bb1(%4: index):
  %5 = arith.cmpi slt, %4, %dim : index
  cf.cond_br %5, ^bb2, ^bb70
```

- `cf.br` 是 control flow dialect 的无条件跳转。
- `^bb1(%4: index)` 是 basic block，`%4` 是 block 参数。
- `arith.cmpi slt` 比较当前循环变量是否小于上界。
- `cf.cond_br` 根据比较结果跳到循环体或退出块。
- 这类结构是 `scf.for` 被降级后的显式控制流形式。

### 2.4 预取仍保留为 `memref.prefetch`

```mlir
memref.prefetch %subview_15[%c0], read, locality<3>, data : memref<?xf32, #map1>
memref.prefetch %subview_16[%c0], read, locality<3>, data : memref<?xf32, #map1>
```

- 第一条预取对应 A 的一维读流。
- 第二条预取对应 B 的一维读流。
- 进入 Arm SME 相关层后，预取仍没有丢失。
- 此时还没有变成 `llvm.intr.prefetch`，因为文件尚未完全降到 LLVM dialect。

### 2.5 masked load 和 vector insert

```mlir
%19 = memref.load %subview_15[%15] : memref<?xf32, #map1>
%20 = vector.insert %19, %16 [%15] : f32 into vector<[16]xf32>
```

- `memref.load` 从 A 或 B 的一维 view 中读取一个标量。
- `vector.insert` 把标量插入 vector 的某个 lane。
- 这说明部分 `vector.transfer_read` 已被展开成显式循环、条件判断、标量 load 和 vector insert。

### 2.6 SME intrinsic

```mlir
%319 = "arm_sme.intr.read.horiz"(...)
"arm_sme.intr.ld1w.horiz"(...)
"arm_sme.intr.mopa"(...)
"arm_sme.intr.st1w.horiz"(...)
```

- `arm_sme.intr.read.horiz` 从 ZA tile 中读取水平 slice。
- `arm_sme.intr.ld1w.horiz` 把内存中的 `f32` 数据加载到 ZA tile 的水平 slice。
- `arm_sme.intr.mopa` 执行 SME matrix outer product accumulate，是矩阵乘核心。
- `arm_sme.intr.st1w.horiz` 把 ZA tile 的水平 slice 存回内存。

### 2.7 为什么还有 `unrealized_conversion_cast`

```mlir
builtin.unrealized_conversion_cast ...
```

- 这是 MLIR lowering 过程中暂时保留的类型桥接。
- 在 `02_arm_sme_prefetch.mlir` 中出现是正常的，因为它不是最终 LLVM dialect 文件。
- 后续 `-test-lower-to-llvm`、`-reconcile-unrealized-casts` 和自定义 repair pass 会继续清理这些桥接。

## 3. `03_llvm_prefetch.mlir`

对应文件：

- [output/03_llvm_prefetch.mlir](/Users/alpaca/Documents/SME/SME1/04_vector_arm_sme_llvm/output/03_llvm_prefetch.mlir)

这是第 04 步最终输出。它已经降到 LLVM dialect，并且可以用 `mlir-translate --mlir-to-llvmir` 转成 LLVM IR `.ll`。

### 3.1 LLVM dialect 函数签名

```mlir
llvm.func @gemm_step4_compute(
  %arg0: !llvm.ptr, %arg1: !llvm.ptr, %arg2: i64, ...
) -> !llvm.struct<(ptr, ptr, i64, array<2 x i64>, array<2 x i64>)>
attributes {arm_locally_streaming, arm_new_za, step4_vector_prefetch = "memref_prefetch"}
```

- `llvm.func` 表示函数已经进入 LLVM dialect。
- `!llvm.ptr` 是 LLVM 指针类型。
- 多个 `i64` 参数来自 memref descriptor 的 offset、size、stride 等字段。
- 返回值是 memref descriptor 结构体。
- `arm_locally_streaming` 和 `arm_new_za` 继续保留 SME 函数属性。

### 3.2 常量、poison 和 descriptor 组装

```mlir
%0 = llvm.mlir.constant(0 : index) : i64
%10 = llvm.mlir.poison : vector<[4]xf32>
%17 = llvm.mlir.poison : !llvm.struct<...>
%18 = llvm.insertvalue %arg14, %17[0] : !llvm.struct<...>
```

- `llvm.mlir.constant` 生成 LLVM dialect 中的常量。
- `llvm.mlir.poison` 生成未定义初值，通常作为 vector 或 struct 的构造起点。
- `llvm.insertvalue` 把参数逐个填入 struct 字段，用来重建返回的 memref descriptor。

### 3.3 vscale 与栈上 tile 缓冲区

```mlir
%25 = "llvm.intr.vscale"() : () -> i64
%26 = llvm.mul %25, %12 : i64
%28 = llvm.alloca %27 x f32 : (i64) -> !llvm.ptr
```

- `llvm.intr.vscale` 读取目标机器的 scalable vector 缩放因子。
- `llvm.mul` 计算运行时 tile 大小。
- `llvm.alloca` 分配 LLVM 层栈内存，用于保存 lowering 后的 tile 缓冲。

### 3.4 LLVM basic block 和分支

```mlir
llvm.br ^bb1(%0 : i64)
^bb1(%52: i64):
  %53 = llvm.icmp "slt" %52, %arg3 : i64
  llvm.cond_br %53, ^bb2, ^bb271
```

- `llvm.br` 是 LLVM dialect 的无条件跳转。
- `^bb1(%52: i64)` 是带参数的 basic block。
- `llvm.icmp "slt"` 是有符号小于比较。
- `llvm.cond_br` 是条件跳转。
- 这类语句共同表示 lowering 后的显式循环控制流。

### 3.5 mask 构造

```mlir
%56 = llvm.intr.stepvector : vector<[16]xi32>
%58 = llvm.trunc %57 : i64 to i32
%59 = llvm.insertelement %58, %14[%15 : i32] : vector<[16]xi32>
%60 = llvm.shufflevector %59, %14 [...]
%61 = llvm.icmp "slt" %56, %60 : vector<[16]xi32>
```

- `llvm.intr.stepvector` 生成 lane 编号向量，例如 `[0, 1, 2, ...]`。
- `llvm.trunc` 把 `i64` 长度截断成 `i32`，供向量 mask 构造使用。
- `llvm.insertelement` 把长度值插入向量第 0 个 lane。
- `llvm.shufflevector` 把该长度值广播到所有 lane。
- `llvm.icmp "slt"` 比较 lane id 是否小于有效长度，生成 mask。

### 3.6 最终预取语句

```mlir
%84 = llvm.getelementptr %82[%83] : (!llvm.ptr, i64) -> !llvm.ptr, f32
"llvm.intr.prefetch"(%84) <{cache = 1 : i32, hint = 3 : i32, rw = 0 : i32}> : (!llvm.ptr) -> ()
```

- `llvm.getelementptr` 计算预取地址。
- `llvm.intr.prefetch` 是 LLVM dialect 的预取 intrinsic。
- `rw = 0` 表示读预取。
- `hint = 3` 表示高局部性。
- `cache = 1` 表示数据 cache。
- 这条预取来自 `01_vector_prefetch.mlir` 中 A 读前的 `memref.prefetch`。

```mlir
%102 = llvm.getelementptr %100[%101] : (!llvm.ptr, i64) -> !llvm.ptr, f32
"llvm.intr.prefetch"(%102) <{cache = 1 : i32, hint = 3 : i32, rw = 0 : i32}> : (!llvm.ptr) -> ()
```

- 这条预取对应 B 读流。
- 两条 `llvm.intr.prefetch` 说明 A/B 两个输入流的预取都保留到了最终 LLVM dialect。

### 3.7 load、insert 和向量构造

```mlir
%92 = llvm.load %91 : !llvm.ptr -> f32
%93 = llvm.insertelement %92, %86[%85 : i64] : vector<[16]xf32>
```

- `llvm.load` 从计算出的 LLVM 指针地址中读取一个 `f32`。
- `llvm.insertelement` 把标量读入结果插入向量 lane。
- 这是 vector transfer lowering 后的显式 LLVM dialect 表示。

### 3.8 SME/SVE intrinsic

```mlir
"arm_sme.intr.ld1w.horiz"(...)
"arm_sme.intr.mopa"(...)
"arm_sme.intr.st1w.horiz"(...)
"arm_sve.intr.psel"(...)
```

- `arm_sme.intr.ld1w.horiz` 对应 SME 水平加载。
- `arm_sme.intr.mopa` 对应 SME 矩阵外积累加，是计算核心。
- `arm_sme.intr.st1w.horiz` 对应 SME 水平存储。
- `arm_sve.intr.psel` 用于处理 predicate select，配合 mask 控制有效 lane。

### 3.9 为什么最终没有 `memref.prefetch`

`03_llvm_prefetch.mlir` 中不再出现 `memref.prefetch`，因为它已经被 lowering 成：

```mlir
"llvm.intr.prefetch"(...)
```

这说明预取语义完成了从 MLIR memref 层到 LLVM dialect 层的转换。

| --- | --- | --- | --- | --- |
| `01_vector_prefetch.mlir` | vector / memref / scf | `memref.prefetch` | `vector.outerproduct` | 证明预取进入 vector 计算主线 |

## 5. 阅读这些文件时应重点观察什么

建议按下面顺序看：

1. 在 `01_vector_prefetch.mlir` 中找 `memref.prefetch`，确认它位于 A/B 的 `vector.transfer_read` 前。
2. 在 `01_vector_prefetch.mlir` 中找 `vector.outerproduct`，确认矩阵乘已经以向量外积表达。
3. 在 `02_arm_sme_prefetch.mlir` 中找 `arm_sme.intr.mopa`，确认计算开始进入 SME 语义。
4. 在 `03_llvm_prefetch.mlir` 中找 `llvm.intr.prefetch`，确认预取进入 LLVM dialect。
5. 在 `03_llvm_prefetch.mlir` 中确认没有 `unrealized_conversion_cast` 或 `arith.index_cast` 残留，说明最终文件已被 repair pass 收口。

一句话总结：第 04 步三个输出文件展示了同一个矩阵乘 kernel 从 vector 预取插入，到 Arm SME 表达，再到 LLVM dialect 预取 intrinsic 的完整主线。
